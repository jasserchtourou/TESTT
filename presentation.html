System Design07 / 14
Design Decision #1: Municipality Lookup
How does "Kirchheim" become a searchable geographic area?
❌ Option A: Exact string match in database
Search 16,000 German municipalities for "Kirchheim". Fast, but fragile. Fails on typos, abbreviations, spacing variations.
✅ Option B: Geocoding with fuzzy matching + caching
Call geocoder (OpenStreetMap) with fuzzy matching. Returns municipality bounds + AGS code. Cache result. 1-2s on first query, instant on repeat.

Why Option B wins: Handles typos and variations. Returns standardized AGS code (used everywhere). Caching eliminates latency on second query.

System Design08 / 14
Design Decision #2: PV Computation Strategy
How do we avoid calling Google Solar API 50,000 times?
❌ Naive Approach: Compute all buildings
50,000 buildings × €0.03/call = €1,500 per briefing. 50,000 ÷ 100 calls/min = 500 minutes (8+ hours). Unacceptable.
✅ Smart Approach: Pre-filter → top 50 only
Filter buildings by roof area (5k qualify). Rank by size. Call Google API only for top 50. Cost: €1.50. Time: 1-2 minutes.

The Insight: A sales engineer cares about the best 10 buildings, not all 50,000. Building geometry is static, so pre-filtering is fast and cheap. Call expensive APIs only for the winners.

System Design09 / 14
Design Decision #3: Where AI Fits
When AI creates value, and when it creates risk
✅ AI: Address fuzzy matching
Match "Rathaus, Kirchheim" against "Town Hall, Kirchheim" in CRM. NLP beats regex.
✅ AI: Narrative generation
Generate contextual summary: "Strong irradiance + existing relationship. Open with Rathaus as anchor." LLMs excel at this.
❌ AI: Financial calculations
Never. LLMs hallucinate numbers. An invented 500 kWp system destroys credibility.

Risk Mitigation: AI writes only text. All numbers come from deterministic code. Pydantic schemas validate every field.

System Design10 / 14
Design Decision #4: CRM Data Join
Linking buildings to deals without a common key
❌ Exact address match
"Hauptstraße 12" vs "Hauptstr. 12" vs "Main Street 12" fail to match. Brittle.
✅ Fuzzy address match + coordinate proximity
Use NLP to compare addresses (handles variations, typos). Also compare lat/lon (catches name variations). Store HubSpot deal metadata in result.

Output: Briefing includes deal context: "2 active deals · Last contact: 3 weeks ago · Key contact: Hans Müller"

System Design11 / 14
V1 Trade-offs: What Ships in 2 Weeks
Shipping means cutting features that don't prove the core idea
❌ Cut: Full city pre-computation
V1: Compute on-demand (2-3 min). V2: Pre-compute solar data at night, serve from cache (instant). V1 proves value faster.
❌ Cut: Bi-directional CRM sync
V1: Read-only CRM data. V2: Push solar scores back to HubSpot deals. Read-only is simpler.
❌ Cut: Multi-step AI agent
V1: One LLM call, one narrative. V2: Agentic loop with brainstorming. Simple is better.
✅ Keep: Core insight
Pre-filter + selective API calls + deterministic math + LLM narrative. If this saves sales engineers 3 hours per municipality, we have product-market fit.
System Design13 / 14
The Core Architectural Insight
Everything flows from one principle

We cannot compute PV for 50,000 buildings in minutes.

But we CAN do this:

1️⃣ Pre-filter 50k buildings to 5k via static geometry
2️⃣ Rank by roof size (biggest = highest ROI)
3️⃣ Call Google Solar API only for top 50
4️⃣ Join with CRM data via fuzzy matching
5️⃣ Generate narrative with LLM (deterministic numbers only)

Result: 2-3 minutes, €1.50 cost, sales-ready briefing

This is the design. Every decision traces back to an assumption or a constraint.

System Design13 / 14
Security & Risk Mitigation
Production-grade safeguards and compliance
🔐 Authentication & Encryption
OAuth2 for HubSpot login. TLS 1.3 encryption in transit. AWS KMS encryption at rest. Rate limiting: 1,000 req/min per user.
🛡️ AI Safety
Pydantic schemas validate every output. AI writes text only. All financial numbers from deterministic code. Circuit breaker for API failures.
📋 Compliance
GDPR: 24-month data retention. Audit logs record every API call. SOC2 ready with CloudWatch + Datadog monitoring.

Risk Principle: Never let AI generate financial numbers. Validation happens at every layer. Auditability is built-in.
