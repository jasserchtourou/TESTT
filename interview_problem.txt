Problem Statement: 
Admi is a B2B SaaS company that helps German municipalities plan and deploy rooftop solar PV systems. Before every sales call, a sales engineer spends 2–4 hours manually pulling together a briefing: how many buildings are in the target municipality, what the aggregate solar potential looks like (roof area, estimated yield, system sizing), whether the municipality is already in the CRM, and any relevant deal history.
Your task is to design a system that automates this briefing. A sales engineer should be able to type or say a municipality name — for example "Gemeinde Kirchheim" — and receive a structured briefing within a few minutes, ready for a call.
An example of output is:
Municipality: Gemeinde Kirchheim · AGS 09175124 · Bayern 
Solar potential 
847 buildings assessed · 312 suitable for solar (37%)
Aggregate system size: ~4.2 MWp · Estimated yield: ~3,800 MWh/year 
Average irradiance: 1,050 kWh/m²/year · Est. CO₂ savings: 1,900 t/year 
Top opportunity 
Rathaus, Hauptstraße 12 — 280 m² south-facing roof, ~42 kWp, ~7.2-year payback 
CRM status 
2 active deals (Potential Analysis stage) · Last contact: 3 weeks ago 
Key contact: Hans Müller, Bürgermeister 
Recommendation 
High-priority lead. Strong irradiance, existing relationship via 2022 energy audit. Recommend opening with Rathaus as anchor project.
Assume you have access to:
A PostGIS database (3dcityDB) with buildings ingested from official CityGML sources
The Google Solar API — given building coordinates, returns roof segmentation and irradiance data (kWh/m²/year) for any location in Germany
A HubSpot CRM instance with full deal and contact history
A React js frontend used by the sales team
The PV calculation should be resolved by AI, the response should looks like:

You have one hour. Start with any clarifying questions, then walk us through your design.
What to Submit
A written design document. Spend no more than 3 hours. We value clear thinking over completeness.
Your document should cover these four sections:
1. Clarifying Assumptions List the assumptions you made that most affect your design. What would you have asked us if this were a live conversation? For each assumption, state what you assumed and why.
2. System Design Describe the architecture: what components exist, what each does, how they communicate, and where state lives. A diagram is welcome but not required. Address at minimum:
How a municipality is identified and looked up.
How and when PV calculations are triggered — not all 50,000 buildings in a municipality can be computed on the fly
How CRM data is joined to building/geospatial data
Suggest a UX for this feature
3. AI in This System Where does AI make this system meaningfully better — and where does it not? Why a simpler non-AI solution would fall short. Also describe one risk specific to using AI here and how you'd mitigate it.
